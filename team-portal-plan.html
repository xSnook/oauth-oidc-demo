Team Portal — v1 Build Plan

An internal web app with Google & Microsoft sign-in, database-managed roles, a user-management screen, and a placeholder dashboard — built with FastAPI, React, and MySQL; developed locally in Docker; deployed to AWS (EC2 + RDS + Route 53) by a GitHub Actions pipeline. This page is the human-readable spec; PLAN.md is the agent-executable version and SETUP.md holds the hands-on walkthroughs.

1Architecture

One EC2 box runs everything app-side under docker compose. Caddy is the only container with public ports — it terminates TLS (automatic Let's Encrypt) and routes /api/* to FastAPI and everything else to an nginx container serving the built SPA. RDS lives in private subnets, reachable only from the instance's security group. Runtime secrets come from SSM Parameter Store, never from GitHub.

TLS note. ACM certificates cannot be installed on a bare EC2 instance — they only attach to managed services (ALB, CloudFront). Caddy + Let's Encrypt is the TLS story here: once DNS resolves to the Elastic IP, the cert issues and renews itself. Set DNS up before the first deploy.

2Authentication flow

The browser gets an ID token from whichever provider the user picks (Google Identity Services or MSAL.js). The backend verifies that token against the provider's published keys — checking the right things for each provider — then finds or creates the user and issues its own session JWT in an HttpOnly cookie. Provider tokens are used once and never stored.

3Data model

Two tables. One human = one users row; each provider login is a row in user_identities, keyed by (provider, provider_subject). Roles are a string column — two static roles don't justify a lookup table yet, and adding one later is a single migration.

5Security decisions (adjudicated — don't relitigate)

These came out of an adversarial review pass over the draft plan; each one closes a concrete attack path or a first-deploy failure.

No email-based account linking

Accounts key on (provider, subject) only. Microsoft email claims are unverified and tenant-admin-controlled — linking by email would let a hostile free Entra tenant capture an existing Google account. Same email via both providers = two accounts, by design.

Tenant-gated admin bootstrap

ADMIN_EMAILS grants admin only at account creation, and only for Google logins (verified email) or Microsoft logins from your tenant (AZURE_ADMIN_TENANT_ID). Otherwise anyone could mint a forged-email Microsoft account and walk in as admin.

Roles enforced from the DB, per request

The session JWT proves identity only — no role claim. Demoting or deactivating a user takes effect on their next request; no stale-token privilege windows.

Microsoft subject = oid, issuer bound to tid

RS256 pinned; audience checked; iss must exactly match the tenant in the (signature-verified) token. sub is pairwise per app registration and would fracture accounts later.

Secrets never touch GitHub or images

CI authenticates via OIDC role assumption (no stored AWS keys); runtime secrets live in SSM Parameter Store; .dockerignore excludes env files so a local build can't bake secrets into a pushed layer.

Login-CSRF guard + SameSite=Lax

State-changing routes are protected by the Lax cookie; the cookie-setting login routes additionally require Content-Type: application/json, closing the cross-site form-post gap.

Locked-out recovery. If the only admin gets demoted/disabled, recovery is a one-line SQL update via the EC2 box — UPDATE users SET role='admin' WHERE email='…'. Documented in PLAN.md §4.3.

7CI/CD pipeline (GitHub Actions YAML)

Push to main runs the full chain; PRs run tests only. AWS access is an OIDC-assumed role scoped to this repo's main — no stored cloud keys.

1test
  • ruff check + format check
  • Alembic single-head check + upgrade head against a real MySQL 8 service container
  • pytest (auth, RBAC, user mgmt)
  • frontend: tsc + eslint + build
→
2build-push
  • assume OIDC deploy role
  • build app-api + app-web (client IDs injected as build args from repo variables)
  • tag git-sha + latest → push to ECR
→
3deploy
  • open port 22 to the runner's IP only (closed again in always())
  • scp deploy/ → /opt/app/; refresh secrets from SSM
  • record rollback tag from the running container
  • alembic upgrade head before the swap · compose up -d
  • smoke test /api/health — deploy fails loudly if unhealthy

Rollback: re-point IMAGE_TAG at .previous_tag and compose up -d. Migrations stay additive so old code always runs on new schema; destructive changes are split across two releases.

8Setup walkthroughs (the parts only you can do)

Full numbered steps live in SETUP.md — these are the maps plus the traps that actually strand people.

1 · Windows 11 Home dev machine ~60–90 min with reboots
  1. WSL2 first — wsl --install in an admin terminal, reboot. Home edition has no Hyper-V, so Docker Desktop requires the WSL2 backend.
  2. Docker Desktop (WSL2 backend) → verify with docker run hello-world. Enable "start when you sign in".
  3. Git for Windows with core.autocrlf=input — CRLF line endings inside Linux images cause env: 'python\r': No such file-style failures.
  4. Node LTS + Python 3.12 (editor tooling only — the app runs in containers). VS Code + Python/Pylance/ESLint/Prettier/Docker extensions.
  5. Clone to C:\dev\…, copy .env.example → .env, fill secrets/client IDs, docker compose up --build, then docker compose exec api alembic upgrade head.
Do not clone inside OneDrive (your Documents folder is synced). OneDrive fights Docker bind mounts and file watchers: laggy hot reload, locked files, endless sync churn.
Port 3306 already taken = an old MySQL Windows service — netstat -ano | findstr ":3306", stop it, or remap the host port.
2 · Google & Microsoft sign-in consoles ~30 min
  1. Google: new Cloud project → OAuth consent screen (External unless you have Workspace; in Testing mode only listed test users can sign in) → OAuth client, type Web application → Authorized JavaScript origins: http://localhost:5173, http://localhost, later https://<domain>. Redirect URIs stay empty — GIS validates origins. Copy the Client ID (the secret is unused).
  2. Microsoft: Entra ID → App registration → account type "any org directory + personal accounts" (must match the common authority) → platform Single-page application with the same URIs → no client secret. Copy the Application (client) ID and the Directory (tenant) ID (the latter gates Microsoft admin bootstrap).
AADSTS9002326 cross-origin token redemption… = the URI is registered under Web instead of SPA. Error 400: origin_mismatch = fix Google's origins list (exact scheme/host/port; edits can take minutes–hours).
"Works for me, not my coworker" on Google = they're not in the test-user list (Testing mode).
3 · GitHub repo & Actions ~15 min
  1. Private repo, push the skeleton, confirm Actions enabled.
  2. Secrets: AWS_ACCOUNT_ID, AWS_ROLE_ARN, EC2_HOST, EC2_SSH_PRIVATE_KEY, EC2_SG_ID. Variables: VITE_GOOGLE_CLIENT_ID, VITE_AZURE_CLIENT_ID.
  3. Optional: branch rule requiring the test check (note: not enforced on Free-plan private repos — the red X is still your signal).
Never create AWS_ACCESS_KEY_ID/SECRET — CI uses OIDC role assumption. Runtime secrets (DB URL, JWT secret) live in SSM, not GitHub.
4 · AWS account, IAM & infrastructure ~60–90 min, in order
  1. Hygiene: root MFA + no root keys → daily-use admin identity (IAM Identity Center or IAM user) → $50/month budget with email alerts.
  2. GitHub OIDC: identity provider token.actions.githubusercontent.com → role app-github-deploy, trust scoped to repo:<org>/<repo>:ref:refs/heads/main; policy = ECR push + temporary SG rule management.
  3. ECR: app-api, app-web (+ keep-last-10 lifecycle).
  4. EC2 role app-ec2-role: SSM core + ECR read + /app/prod parameter read (Resource must include both the path ARN and /* — GetParametersByPath authorizes against the path itself).
  5. EC2 t3.small Ubuntu 24.04 x86: SG app-ec2-sg = 80/443 only, no port 22; key pair app-deploy-key (pipeline-only); user-data installs Docker + compose, AWS CLI v2 via the zip installer, MySQL client, and Docker log rotation. Attach an Elastic IP; bootstrap /opt/app/.env.
  6. RDS app-prod-mysql: db.t4g.micro, Public access No, SG app-rds-sg sourced from app-ec2-sg, initial DB appdb, utf8mb4 parameter group, 7-day backups, deletion protection. Create the app's appuser from the box.
  7. SSM: the nine /app/prod/* parameters; verify from the instance with get-parameters-by-path (the same call the deploy uses).
  8. Route 53: buy the domain (or delegate NS records), apex A record → Elastic IP, before the first deploy so Caddy can get its certificate. Skip www in v1. Then the two return trips: Google origins + Entra redirect URIs.
Don't install the AWS CLI with snap in user-data — it races cloud-init ("device not yet seeded") and lands off the default PATH. The zip installer to /usr/local/bin is the reliable route.
Restoring an RDS snapshot creates a new instance with a new endpoint and default SG/parameter group — re-attach both and update /app/prod/DATABASE_URL.
